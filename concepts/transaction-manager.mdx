title: 'Transaction manager'
description: 'Execute, retry and finalize transactions with strong guarantees'
---

Kwala manages the full transaction lifecycle when your workflow calls a smart contract function. It handles gas estimation, submission, retries, and confirmation so your transactions land on-chain reliably.

## Calling smart contract functions

Use the `call` action type to interact with deployed smart contracts.

```yaml
actions:
  - Name: transfer_funds
    Type: call
    TargetContract: 0x0f51bb10119727a7e5ea3538074fb341f56b09ad
    TargetFunction: function transfer(address recipient, uint256 amount)
    TargetParams:
      - re.event(0)    # recipient address from triggering event
      - 100            # transfer amount
    ChainID: 1
    RetriesUntilSuccess: 5
```

### Configuration fields

| Field | Required | Description |
|-------|----------|-------------|
| **TargetContract** | Required | Address of the smart contract to interact with |
| **TargetFunction** | Required | Function signature to call (e.g., `function transfer(address, uint256)`) |
| **TargetParams** | Required | Parameters to pass to the function |
| **ChainID** | Required | Blockchain network where the contract is deployed |
| **RetriesUntilSuccess** | Optional | Number of retry attempts if the call fails (default: 0) |

## Transaction finalization guarantees

Kwala provides strong guarantees that your transactions will finalize:

### Automatic retries

The `RetriesUntilSuccess` field ensures critical operations complete even when network conditions cause temporary failures.

```yaml
actions:
  - Name: critical_transfer
    Type: call
    TargetContract: 0x65b2b850fee0e0c13d956ab5...
    TargetFunction: function execute(address to)
    TargetParams:
      - 0x742d35Cc6e9A5f1e7A0e6FD4C8b2Bc3F5d9E1234
    ChainID: 80002
    RetriesUntilSuccess: 5
```

**Retry behavior:**
- Retries happen automatically on failure
- A delay is applied between retries
- Total attempts = 1 (initial) + RetriesUntilSuccess

### Using dynamic parameters

Reference event data in your transaction parameters using `re.event()`:

```yaml
TargetParams:
  - re.event(0)    # first parameter from triggering event
  - re.event(1)    # second parameter
```

This enables workflows where the triggering event determines transaction details.
